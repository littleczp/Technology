# 高并发

<details>

<summary>请求量对服务器压力较大，协议上有什么优化空间</summary>

<mark style="color:red;">不要直接罗列优化点，而是先拆解问题</mark>

先判断压力来源：是请求次数太多、单次请求太重、轮询频繁、重复请求多、连接建立成本高，还是失败重试放大了流量。

然后从协议层分别做

1. 提供批量化接口（请求次数太多），但要控制batch大小、超时和部分失败策略
2. 返回增量数据、字段裁剪（单次请求太重）
3. 长连接或推送（轮询频繁），弱实时拉长轮询时间/指数退避
4. 请求去重与幂等设计，限流和退避等优化、RequestID（失败重试）



还有什么优化空间？非业务本身层面

1. 压缩与更紧凑的编码：HTTP可以使用gzip压缩、内部RPC可以考虑使用protobuf取代json
2. 长连接、HTTP/2 多路复用、连接池，减少 TLS 握手和 TCP 建连开销。但长连接会占用服务端连接资源，所以要配合心跳、空闲超时和连接数治理。

</details>
